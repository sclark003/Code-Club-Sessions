üß™ Testing Your Game

Today we'll thoroughly test your game, find and fix any problems, balance the difficulty, and prepare it for sharing with others!

üöÄ Activity 1: Systematic Testing

Step 1: Create a testing checklist:
  • Start menu works
  • Game starts correctly
  • Player movement is smooth
  • All buttons work
  • Collisions work correctly
  • Scoring works
  • Lives system works
  • Power-ups work
  • Enemies behave correctly
  • Boss battles work
  • Level progression works
  • Game over screen works
  • High score saves
  • Pause works
  • All sounds play
Step 2: Test each item on your checklist
Step 3: Write down any problems you find

üêõ Activity 2: Finding Bugs

Step 1: Common bugs to look for:
  • Sprites disappearing
  • Movement not working
  • Collisions not detecting
  • Score not updating
  • Game freezing
  • Sprites going off screen
  • Sounds not playing
  • Level not progressing
Step 2: Test edge cases:
  • What happens at screen boundaries?
  • What if player collects item very quickly?
  • What if multiple enemies hit at once?
  • What if score gets very high?
Step 3: Try to break your game - find ways it might fail

üîß Activity 3: Fixing Bugs

Step 1: When you find a bug:
  • Write down what happened
  • Note what you were doing when it happened
  • Try to reproduce it (make it happen again)
  • Find the code that causes it
  • Fix the problem
  • Test to make sure it's fixed
Step 2: Debugging tips:
  • Add temporary text to see what's happening
  • Check variable values
  • Test code in small pieces
  • Comment out code to isolate problems

‚öñÔ∏è Activity 4: Balancing Difficulty

Step 1: Play through your game and note:
  • Is it too easy? (boring)
  • Is it too hard? (frustrating)
  • Is it just right? (challenging but fun)
Step 2: Adjust difficulty:
  • Too easy: Increase enemy speed, add more enemies, reduce lives
  • Too hard: Slow enemies, add more power-ups, increase lives
  • Make early levels easier, later levels harder
Step 3: Test with different skill levels (beginners and experienced players)

üë• Activity 5: Playtesting

Step 1: Have others play your game
Step 2: Watch them play (don't help unless they're stuck)
Step 3: Note:
  • Where do they get confused?
  • What do they enjoy?
  • What's frustrating?
  • Do they understand the controls?
  • Is the goal clear?
Step 4: Ask for feedback:
  • What did you like?
  • What would you change?
  • Was it too easy/hard?
  • Would you play it again?

üìù Activity 6: Final Checklist

Step 1: Before sharing, make sure:
  • Game is fully playable from start to finish
  • No major bugs
  • Difficulty is balanced
  • All features work
  • Instructions are clear
  • Game is fun to play

üåü Activity 7: Sharing Your Game

Step 1: In MakeCode Arcade, click "Share"
Step 2: Options:
  • Share link (others can play in browser)
  • Download as .hex file (for Arcade console)
  • Embed code (for websites)
Step 3: Create a game description:
  • Game title
  • What type of game it is
  • How to play
  • What makes it special
Step 4: Share with:
  • Classmates
  • Friends and family
  • Online communities
  • Game jams or competitions

Step 1: You've created a complete game! That's amazing!
Step 2: Present your game:
  • Show it to the class
  • Explain how it works
  • Share what you learned
  • Talk about challenges you overcame
Step 3: Reflect on your journey:
  • What was your favorite part?
  • What was most challenging?
  • What would you do differently?
  • What will you make next?
Complete final testing and polish:
  • Test every feature thoroughly
  • Fix all bugs

  • Get feedback from at least 3 different players
  • Make improvements based on feedback
  • Create a shareable version
  • Write a game description

üí° Key Concepts

  • Testing: Systematically checking that everything works
  • Bug: A problem or error in the code
  • Debugging: Finding and fixing bugs
  • Balance: Making difficulty appropriate (not too easy or hard)
  • Playtesting: Having others play your game to get feedback
  • Feedback: Comments and suggestions from players
  • Iteration: Making improvements based on testing and feedback
